# v1.4.0 结构化索引实施与验收结果

验收日期：2026-07-17  
实施基线：`5073f9f899e83759a6e21ec1ea56a7ae282f84eb`（v1.3.5）

## 已实现

- 独立 Pydantic 领域模型：`CodeEntity`、`PaperEntity`、`KnowledgeEdge`、`EvidenceRef`、`SymbolChunk`、`IndexedFile`、`IndexManifest`。
- 显式与 task-scoped 仓库身份、跨平台安全路径、模块根、稳定 Entity/Edge ID、内容哈希和完整版本化 `input_hash`。
- 候选列表式 Symbol Table、import alias/from/relative resolver、继承、实例化、局部调用、`self.method()`、可确定的 `self.module(x) → forward` 和 unresolved/ambiguous 保留。
- 从旧 AST、文件/函数/模型规则事实和规则论文对齐增量构建实体、关系、证据和 Symbol-aware Chunk，不消费 LLM/VLM 解释。
- 独立 `data/structured_index.sqlite3`、编号 migration、`indexed_files` 与 `symbol_chunks`、repo lease、staging、短事务原子激活、版本回滚和 `index_manifest.json`。
- `structured_index_build` 固定在 `paper_code_align` 之后、所有 LLM/VLM 增强之前；默认 feature flag 关闭，失败不阻断旧报告。
- API、CLI 和任务结果增加可选索引参数/摘要；未改动前端源码或旧报告生成逻辑。

## 数据库与 API 变化

- 新数据库 schema version：`PRAGMA user_version=1`。
- 新表：`repositories`、`index_versions`、`indexed_files`、`code_entities`、`paper_entities`、`knowledge_edges`、`evidence_refs`、`symbol_chunks`。
- 现有任务创建 API 增加可选字段 `structured_index_enabled`、`repository_key`、`structured_index_db_path`；任务结果增加可选 `index_manifest`。
- 未新增索引查询路由；旧 API 字段和前端类型无需修改。

## 兼容性

- feature flag 关闭时不生成 manifest、不使用结构化索引数据库。
- 开启/关闭索引得到的 `parsed_files.json`、`file_analysis.json`、`function_analysis.json` 结构和规范化语义一致；报告将任务特有绝对仓库路径规范化后内容一致。
- `parsed_files.json`、`file_analysis.json`、`function_analysis.json`、旧报告和前端继续独立工作；不承诺 JSON 字节、空白或键顺序相同。
- 前端目录按计划保持不变，因此其 npm 包元数据仍为 1.3.5；后端应用与项目版本已更新为 1.4.0。

## 已知限制

- Python 动态调用、反射、monkey patch、Registry/Factory 和运行时 import 不强行解析，保留 unresolved/ambiguous。
- 条件/异常作用域重复定义不会自动选择；无条件重复定义按后定义覆盖。增删同名声明仍可能造成 declaration ordinal 漂移。
- 文件移动按删除旧路径实体并创建新实体处理，不推断 Git rename。
- superseded 历史版本默认保留，当前仅提供持久化层显式回滚能力，没有 retention API 或前端。
- 大仓库仍在单进程内构建内存对象并写 staging；尚未实现分片并行或流式激活。
- v1.4.0 不包含 Hybrid RAG、向量数据库、动态 Agent、Trace/Bad Case 前端或多 Agent。
