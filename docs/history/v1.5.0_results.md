# v1.5.0 实施与验收结果

验收日期：2026-07-18  
开发基线：`04ea17e v1.4:结构化索引基础开发`  
分支：`upgrade/v1.4-structured-index`

## 已实现范围

- 独立 Retrieval/Public API/Context/Answer Pydantic Schema。
- 只读 v1.4 Snapshot、Entity、Chunk、Evidence 和 Graph 查询边界。
- SQLite FTS5 generation、ready 原子切换、失败隔离、exact symbol/path boost。
- 可选 FastEmbed/Qdrant Local Dense、版本化 Point ID、full profile hash 校验和同步 Manifest。
- 可选 Qdrant BM25 Sparse；缺失时 FTS5 仍是完整 fallback。
- Dense/Sparse Preliminary RRF、Graph seed/一二跳扩展、EntityChunkSelector、Final RRF。
- 保留 Hybrid 信号的 profile-aware Reranker 融合和失败回退。
- 40% 单项上限、保守代码 token 估算和 Provider 前确定性缩减的 Context Builder。
- 固定单轮 Research Query、显式外部文本授权和 Citation Validator。
- 始终注册、flag 关闭返回 `503 retrieval_disabled` 的三个 API。
- 30 Dev + 10 Locked Test、可重建结构化 fixture、指标与五种消融运行模式。

未修改 v1.4 Entity/Edge/Evidence/Chunk ID 算法、现有 LangGraph 节点顺序、旧 JSON、旧
报告和前端类型。

## FTS5 Sparse 实测基线

使用 `evaluation/retrieval/build_fixture.py` 重建固定 v1/v2 Snapshot，并运行：

```text
scripts/run_retrieval_benchmark.py --mode sparse_only
```

这是本机热缓存附近的一次真实执行，不是质量承诺：

| Split | Recall@1 | Recall@5/10 | MRR | nDCG@5/10 | 平均延迟 | P95 |
| -- | --: | --: | --: | --: | --: | --: |
| Dev（30） | 0.4833 | 0.6167 | 0.5833 | 0.5809 | 1.44 ms | 2.30 ms |
| Locked Test（10） | 0.3500 | 0.5000 | 0.5500 | 0.4857 | 1.28 ms | 2.00 ms |
| All（40） | 0.4500 | 0.5875 | 0.5750 | 0.5571 | 1.40 ms | 2.27 ms |

6 条 `exact_symbol` case 的 Recall@5 为 100%。Sparse-only 不执行 Graph，因此仅对包含
gold graph path 的 case 统计的 Graph Path Recall 为 0；无 Graph gold 的 case 不再被当作
成功样本抬高该指标。

## 未执行的手动验收

本轮没有安装可选 Qdrant/FastEmbed 依赖，没有下载 Embedding/Reranker 模型，也没有调用
收费 Provider。因此以下结果必须在显式预缓存模型并确认 License/revision 后补录：

- 多语言 MiniLM 和 Jina code 的真实 Dense Recall/延迟消融。
- Qdrant BM25 Sparse 与 FTS5 的对比。
- BGE/轻量 Cross Encoder 的真实 Reranker 收益和 P95 延迟。
- Dense-only、Dense+Sparse、+Graph、+Reranker 的完整 Locked Test 数值。

代码和自动测试已覆盖无模型、无网络 fallback，不把未运行的真实模型实验宣称为通过。
